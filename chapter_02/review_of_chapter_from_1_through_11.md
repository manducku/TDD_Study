###Chapter 03 - 단위 테스트를 이용한 간단한 홈페이지 테스트

 - 기능 테스트 : 사용자 관점의 새로운 기능성을 정의하는 것부터 시작한다.
 - 기능 테스트 실패 후, 테스트를 통과할 코드를 생각한다. 하나 또는 그 이상의 단위 테스트를 이용해서 어떻게 코드가 동작하는지 정의( 기본적으로 모든 코드가 하나 이상의 단위 테스트에 의해 테스트 돼야 한다 )
 - 단위 테스트 실패 후, 단위 테스트를 통과할 수 있을 정도의 최소한의 코드만 작성. 기능 테스트가 완전해질 때까지 이전, 지금 과정을 반복해야 할 수도 있다
 - 기능 테스트 재실행, 통과하는지 확인. 이 때 새로운 단위 테스트를 작성해야 할 수도 있다.

> 이 장에서는 기본적인 장고 뷰를 테스트하는 과정을 보여준다. 낯섬의 연속이지만 곧 익숙해 진다.

 - `단위 테스트-코드 주기`
  1. 터미널에서 단위 테스트를 실행해서 어떻게 실패하는지 확인
  2. 편집기상에서 현재 실패 테스트를 수정하기 위한 최소한의 코드 변경
  3. 1,2 과정을 반복한다
 - 위 과정에서 코드품질을 높이고 싶다면 코드 변경을 최소화해야 한다.
 - 최소화된 코드는 하나하나 테스트에 의해 검증돼야 한다.

###Chapter 04 - 왜 테스트를 하는 것인가?

> 여러 의혹이드는 시기다. 4장에서는 이런 의혹을 설명해준다.
> TDD의 장점 중 하나는 다음에 무엇을 해야 할지 잊어버릴 걱정이 없다는 것이다. 테스트를 실행하기만 하면 다음 작업이 무엇인지 가르쳐 준다.

 - "상수는 테스트하지 마라", 템플릿을 사용하기 위한 리팩터링(기능은 바꾸지 않고 코드 자체를 개선)
 - p54 ~ p55 의 `TDD 프로세스`를 참고하자

###Chapter 05 - 사용자 입력 저장하기

 - `레드/그린/리팩터`
  - 실패할 단위 테스트를 작성함으로써 작업을 시작(레드)
  - 이 테스트를 통과할 최소 코드를 작성한다(그린). 편법이라도 상관없다.
  - 코드를 리팩터링해서 이해할 수 있는 코드로 만든다.


 - `Three strikes and Refactor`, `DRY(Don't Repeat Yourself)`
 - 한 번 정도는 복붙을 봐줄 수 있지만, 같은 코드가 세 번 등장하면 중복을 제거해야 한다.
 - 테스트에서도 `헬퍼 메서드`를 이용한다.
 - 작업 메모장을 활용하여 몇몇 작업은 미룬다. 문제를 바로 해결하는 게 좋을 때가 있고 문제가 너무 크면 다른 방법을 생각해야 하는 경우가 있다.
 - 좋은 단위 테스트는 하나의 기능만 테스트 해야 한다
